Módulo 4 · Next.js

Dados & deploy

FETCH, BUILD ESTÁTICO E GITHUB PAGES

Meta final

Consumir dados, finalizar a interface e publicar em produção.

Agenda do módulo

  • Fetch assíncrono em Server Components.
  • Conceito de renderização estática.
  • Benefícios de next/image.
  • Variáveis de ambiente em alto nível.
  • Build estático (output: 'export') e deploy gratuito no GitHub Pages.

Server Component com async fetch

Você pode fazer fetch diretamente no componente de página com async/await.

export default async function Page() { const data = await fetch(url).then(r => r.json()); return <main>...</main>; }

Renderização estática

Com output: 'export' no Next.js, o build gera HTML/CSS/JS puros na pasta out/ — ideal para GitHub Pages.

next.config.ts
const nextConfig = {
  output: "export",  // gera pasta out/ com arquivos estáticos
  trailingSlash: true, // ajuda rotas no GitHub Pages
};
  • Sem servidor Node em produção — só arquivos estáticos.
  • Hospedagem gratuita no GitHub Pages.

next/image na prática

O componente de imagem do Next otimiza tamanho, formato e carregamento.

  • Lazy loading por padrão em muitos cenários.
  • Ajuda a melhorar métricas de performance visual.
  • Controle de dimensões evita layout shift.

Variáveis de ambiente

Dados sensíveis vão em .env.local, nunca hardcoded no código.

NEXT_PUBLIC_* expõe ao cliente.

Sem prefixo, a variável fica só no servidor.

Build estático

Com as variáveis de ambiente certas, o build gera a pasta out/ pronta para publicar.

terminal
# Linux / macOS
GITHUB_PAGES=true GITHUB_REPOSITORY=seu-usuario/fluxolab-app npm run build

# Resultado: pasta out/ com index.html, precos/, contato/...

Teste local: npx serve out e abra a URL indicada.

Deploy no GitHub Pages

Hospedagem gratuita para sites estáticos — perfeita para o FluxoLab.

  • Repositório no GitHub com o código do app.
  • GitHub Actions faz o build e publica automaticamente.
  • URL final: https://seu-usuario.github.io/nome-do-repo/

Repositório em subpasta do site exige basePath no next.config.ts.

Boas práticas pré-release

  • Testar rotas principais e páginas secundárias.
  • Revisar metadados de título/descrição.
  • Verificar imagens remotas configuradas.
  • Confirmar fallback para rota inexistente.

Ambientes: dev, preview e produção

  • Dev local: ciclo rápido para construir features.
  • Preview: revisão antes de publicar oficialmente.
  • Produção: versão estável para utilizadores finais.

Mantenha o mesmo checklist de validação entre preview e produção.

Projeto guiado

Módulo 4 focado em deploy: do next.config até o site online no GitHub Pages — 20 passos.

Deploy no GitHub Pages

Passo a passo literal — do next.config até o site online.

Passo 1/20 · Visão geral do fluxo

O deploy tem 4 fases. Siga nesta ordem — pular uma fase costuma causar erro.

  1. A · Configurar next.config.ts para export estático + basePath
  2. B · Testar build local (gerar pasta out/ e abrir no navegador)
  3. C · Criar workflow + enviar código para o GitHub (git push)
  4. D · Ativar GitHub Pages e aguardar a Action publicar o site

npm run dev continua igual no dia a dia — as mudanças abaixo só afetam npm run build com variáveis de ambiente.

Passo 2/20 · Entender a URL final

Repositórios de projeto usam subpasta no domínio do GitHub:

https://SEU-USUARIO.github.io/NOME-DO-REPO/

Exemplo: usuário maria-dev, repo fluxolab-app

→ https://maria-dev.github.io/fluxolab-app/

  • Home: /fluxolab-app/
  • Preços: /fluxolab-app/precos/
  • Contato: /fluxolab-app/contato/

Por isso o Next precisa de basePath: "/fluxolab-app" no build de produção.

Passo 3/20 · Substituir next.config.ts inteiro

Abra next.config.ts na raiz do projeto e substitua todo o conteúdo por este arquivo:

next.config.ts (arquivo completo)
import type { NextConfig } from "next";

const isGithubPages = process.env.GITHUB_PAGES === "true";
const repoName = process.env.GITHUB_REPOSITORY?.split("/")[1];
const basePath =
  isGithubPages && repoName ? `/${repoName}` : "";

const nextConfig: NextConfig = {
  ...(isGithubPages ? { output: "export" as const } : {}),
  basePath: basePath || undefined,
  assetPrefix: basePath ? `${basePath}/` : undefined,
  trailingSlash: isGithubPages,
  images: {
    unoptimized: isGithubPages,
    remotePatterns: [
      { protocol: "https", hostname: "images.unsplash.com" },
    ],
  },
};

export default nextConfig;

Salve o arquivo. Não precisa reiniciar o npm run dev por causa disso.

Passo 4/20 · O que cada linha do next.config faz

  • GITHUB_PAGES === "true" — liga modo GitHub Pages só no build de produção.
  • GITHUB_REPOSITORY — Actions envia usuario/repo; extraímos o nome do repo.
  • output: "export" — gera HTML estático na pasta out/ (obrigatório no Pages).
  • basePath + assetPrefix — CSS/JS/links funcionam em /nome-do-repo/.
  • trailingSlash: true — URLs com barra final; evita 404 no Pages.
  • images.unoptimized — export estático não tem servidor de imagens do Next.

Passo 5/20 · Checklist antes do build

  • Você está na pasta fluxolab-app (onde está o package.json).
  • npm run dev funcionava sem erros antes deste passo.
  • Já definiu o nome do repositório GitHub (ex.: fluxolab-app).
  • Anotou seu usuário GitHub (ex.: maria-dev).

Nos comandos abaixo, troque seu-usuario e fluxolab-app pelos seus valores reais.

Passo 6/20 · Build no Windows · preparação

1–3. Abra o PowerShell na pasta do projeto, confirme package.json e defina as variáveis:

PowerShell
cd C:\caminho\para\fluxolab-app
$env:GITHUB_PAGES = "true"
$env:GITHUB_REPOSITORY = "seu-usuario/fluxolab-app"

As variáveis valem só nesta janela do terminal.

Passo 7/20 · Build no Windows · executar

4. Rode o build de produção:

npm run build

Sucesso se no final aparecer algo como:

  • ✓ Compiled successfully
  • ✓ Generating static pages
  • Pasta out criada na raiz do projeto

Se der erro, leia a mensagem — geralmente é typo em algum componente ou import faltando.

Passo 8/20 · Validar pasta out/ e servir

5. Verifique a pasta out/:

dir out

Deve existir: index.html, pasta precos/, pasta contato/, pasta _next/.

6. Sirva localmente e teste:

npx serve out

Abra no navegador: http://localhost:3000/fluxolab-app/ (troque fluxolab-app pelo nome do seu repo).

Passo 9/20 · Build no Linux / macOS

Mesma lógica do Windows — sintaxe diferente para variáveis:

terminal (bash/zsh)
cd ~/caminho/para/fluxolab-app
GITHUB_PAGES=true GITHUB_REPOSITORY=seu-usuario/fluxolab-app npm run build
ls out
npx serve out

URL de teste: http://localhost:3000/fluxolab-app/

Passo 10/20 · Validar build localmente

Com o serve rodando, clique e confira:

  • Home carrega com CSS (não página branca sem estilo)
  • Link Preços abre a página de planos
  • Link Contato abre o formulário
  • Ícones Font Awesome aparecem no header
  • Imagem do Hero carrega

Se algo falhar aqui, corrija antes de subir pro GitHub — o erro online será o mesmo.

Passo 11/20 · Criar repositório no GitHub · 1/2

  1. 1. Acesse github.com e faça login.
  2. 2. Clique no + (canto superior direito) → New repository.
  3. 3. Em Repository name, digite: fluxolab-app (ou o nome que usou no build).
  4. 4. Deixe Public selecionado.

Passo 12/20 · Criar pasta do workflow

No VS Code, dentro de fluxolab-app:

  1. 1. Clique com botão direito na raiz do projeto → New Folder → nome: .github
  2. 2. Dentro de .github, crie pasta workflows
  3. 3. Dentro de workflows, crie arquivo deploy-pages.yml

fluxolab-app/.github/workflows/deploy-pages.yml

Passo 13/20 · Arquivo deploy-pages.yml completo

Cole todo o conteúdo abaixo no arquivo — inclusive o job deploy no final:

.github/workflows/deploy-pages.yml
name: Deploy estático no GitHub Pages

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: pages
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - name: Instalar dependências e gerar build estática
        env:
          GITHUB_PAGES: "true"
          GITHUB_REPOSITORY: ${{ github.repository }}
        run: |
          npm ci
          npm run build

      - uses: actions/upload-pages-artifact@v3
        with:
          path: out

  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/deploy-pages@v4
        id: deployment

Salve o arquivo. A linha ${{ github.repository }} é preenchida automaticamente pelo GitHub — não troque.

Passo 14/20 · O que o workflow faz

  • on: push main — cada push na branch main dispara deploy automático.
  • workflow_dispatch — botão “Run workflow” manual na aba Actions.
  • Job build — clona repo, instala deps, roda npm run build com GITHUB_PAGES=true, envia pasta out/.
  • Job deploy — publica o artefato no GitHub Pages (só roda se build passar).

Passo 15/20 · git init no projeto

No terminal, dentro de fluxolab-app:

git init

Resposta esperada: Initialized empty Git repository...

Se já tinha git antes, pode pular — ou apague a pasta .git só se souber o que está fazendo.

Passo 16/20 · git add e commit

git add .
git status
git commit -m "FluxoLab — landing Next.js com deploy"

Em git status, confira se aparecem:

  • .github/workflows/deploy-pages.yml
  • next.config.ts
  • src/ (componentes e páginas)

node_modules/ e .next/ não devem entrar (já estão no .gitignore).

Passo 17/20 · Branch main e remote

git branch -M main
git remote add origin https://github.com/SEU-USUARIO/fluxolab-app.git

Troque a URL pela que você copiou no Passo 11.

Se der “remote origin already exists”: git remote set-url origin URL-AQUI

Passo 18/20 · git push e autenticação

git push -u origin main

O GitHub vai pedir login. Opções comuns:

  • GitHub CLI: gh auth login antes do push
  • HTTPS: use Personal Access Token como senha (não a senha da conta)
  • VS Code: popup de login do GitHub ao fazer push

Sucesso: branch 'main' set up to track 'origin/main'

Passo 19/20 · Ativar GitHub Pages e Actions

  1. 1. No repositório: Settings → Pages
  2. 2. Em Build and deployment, escolha GitHub Actions como fonte
  3. 3. Aba Actions → workflow Deploy estático no GitHub Pages → aguarde ficar verde
  4. 4. Volte em Pages e copie a URL publicada (usuario.github.io/repo/)

Erros comuns: workflow não encontrado (confira .github/workflows/deploy-pages.yml), build falhou (veja o log na Action), CSS ausente (confira basePath).

Passo 20/20 · Checkpoint final

  • Home, Preços e Contato online com estilo e ícones.
  • URL pública no formato *.github.io/nome-do-repo/.
  • Action “Deploy estático no GitHub Pages” verde na aba Actions.
  • Novo push na main republica o site automaticamente.
